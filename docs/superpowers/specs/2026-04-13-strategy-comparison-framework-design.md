# Strategy Comparison Framework — Design

**Date:** 2026-04-13
**Status:** Draft for review
**Scope:** v1 — preset-based strategy comparison on the existing `/backtest` page

## Motivation

Over the last two weeks, the signal generation, allocation, and risk pipelines have each been iterated through 3+ variants (original scoring → regime filter → HMM + top-3; flat → regime-dynamic → risk-weighted; no breakers → daily+streak → full pipeline). A new VWAP + EMA + Bollinger entry is also on the roadmap. Today there is no way to compare iterations side-by-side — the current production path is whatever last shipped. Users (including Zaky) should be able to answer: **"which of these strategies would have performed best over the last N bars?"**

## Goals

1. Define a single `Strategy` interface that cleanly encapsulates entry logic, allocation, and risk.
2. Ship **5 named presets** that represent the real iterations the codebase has been through.
3. Extend the existing `/backtest` page to compare multiple presets side-by-side on the same candle window.
4. Keep live signal generation unchanged — one active preset at a time, swappable via env var.
5. Preserve existing production behavior: default preset must reproduce today's signals byte-for-byte (or with a documented, intentional delta).

## Non-Goals (v1)

- Custom strategy builder UI — deferred to v2 once we see which knobs users actually turn.
- Arbitrary date-range backtests — blocked by Market Data Hub (MDH) only supporting `limit`, not `from`/`to`.
- Running all presets live in parallel — expensive, deferred.
- Saving, sharing, or versioning comparison results.
- Per-symbol preset overrides.

## Architecture

Hybrid: entry logic is a plugin (one module per entry strategy), allocation and risk are config variants dispatched into the existing allocator and risk pipeline.

### Strategy interface

New package: `packages/strategies/` (or nested under `packages/signals/src/strategies/` if the monorepo prefers fewer packages — decide during implementation planning).

```ts
interface Strategy {
  id: 'classic' | 'regime-aware' | 'hmm-top3' | 'vwap-ema-bb' | 'full-risk';
  name: string;
  description: string;
  entry: EntryModule;
  allocation: AllocationConfig;
  risk: RiskConfig;
}

interface EntryModule {
  id: string;
  generateSignals(candles: OHLCV[], context: EntryContext): Signal[];
}

type AllocationConfig =
  | { kind: 'flat' }
  | { kind: 'regime-dynamic' }
  | { kind: 'risk-weighted' };

type RiskConfig =
  | { kind: 'none' }
  | { kind: 'daily-streak' }
  | { kind: 'full-pipeline' };
```

- **Entry = plugin.** Four modules under `packages/strategies/entry/`: `classic.ts`, `regime-aware.ts`, `hmm-top3.ts`, `vwap-ema-bb.ts`. Each exports an `EntryModule`. The first three wrap existing signal-gen code; `vwap-ema-bb.ts` is new.
- **Allocation / Risk = config.** The existing allocator (`packages/signals/src/allocation/allocator.ts`) and risk pipeline already know these modes — `AllocationConfig` and `RiskConfig` just select which path to run.
- Adding a 6th strategy later = add one file under `entry/` + one entry in `presets.ts`.

### Presets (v1)

| ID | Entry | Allocation | Risk | Purpose |
|---|---|---|---|---|
| `classic` | Original scoring | flat | none | Baseline |
| `regime-aware` | Regime filter (commit `95ff3fc4`) | regime-dynamic | daily-streak | First regime iteration |
| `hmm-top3` | HMM + top-3 (commit `bbcb39f6`) | regime-dynamic | daily-streak | **Default — current production** |
| `vwap-ema-bb` | VWAP + EMA + Bollinger (new) | flat | daily-streak | New indicator-based entry |
| `full-risk` | HMM + top-3 | risk-weighted | full-pipeline | Most conservative |

Presets live in `packages/strategies/presets.ts` as `const` objects and are indexed by `id`.

### Default preset compatibility

The `hmm-top3` preset MUST reproduce today's production signals. During implementation, wire it first and run a comparison against a snapshot of the current signal log. Any delta is a bug and must be resolved before other presets are added.

## Data Flow

### Backtest path

1. User loads `/backtest`.
2. User picks: symbol, timeframe (M15 / H1 / H4 / D1), candle limit (default 300, max 1000), and one or more presets via a multi-select.
3. Client calls `fetchHubCandles(symbol, timeframe, limit)` — existing function in `apps/web/app/lib/data-providers/market-data-hub.ts`. One fetch, shared across all selected presets.
4. For each selected preset: `runBacktest(candles, preset) → BacktestResult`. Executed via `Promise.all`.
5. Results rendered in comparison UI (see below).

### Live path

Unchanged. The signal engine cron reads `process.env.SIGNAL_ENGINE_PRESET` (defaults to `hmm-top3`) and invokes the corresponding preset's entry module + allocator + risk pipeline. The resulting signal records carry a `strategyId` field so the UI can display a "Generated by: HMM Top-3" badge.

## Backtest Engine Changes

Current state: [apps/web/app/backtest/page.tsx](apps/web/app/backtest/page.tsx) is a client-side simulator with equity curves, price/indicator charts, trade list, and metrics. It runs one hardcoded strategy inline.

### New: `runBacktest(candles, strategy)`

Extracted into `apps/web/app/backtest/run-backtest.ts` (or `packages/strategies/backtest.ts` if we want it reusable). Takes candles + a `Strategy`, returns the existing `BacktestResult` shape plus `strategyId`.

### Comparison mode

- **Preset multi-select** at the top of the page — checkboxes, all 5 presets visible, default = `hmm-top3` only.
- **Equity curve overlay** — single chart, one line per selected preset, distinct colors, legend.
- **Metrics table** — rows = presets, columns = Total Return, Win Rate, Profit Factor, Max Drawdown, Sharpe Ratio, Trade Count. Per-column "best" cell highlighting (green for max, with Max Drawdown inverted).
- **Single-preset drilldown** — clicking a row in the metrics table (or deselecting all but one) reverts to the existing single-strategy view (price chart, indicators, trades list).

### Symbol + limit picker

- Symbol: dropdown populated from the symbol universe already used elsewhere in the app.
- Timeframe: M15 / H1 / H4 / D1 (matching MDH's supported intervals).
- Limit: numeric input, 50–1000, default 300. UI shows the approximate time window ("≈ 12 days at H1").

## Error Handling

- **MDH fetch failure** — banner at top of page: "Market data unavailable — try again." Do not crash the comparison view. Previous results remain visible.
- **Individual strategy throws during backtest** — that preset's row in the metrics table shows "Failed" with the error message as a tooltip. Other presets still render. The equity chart omits the failed line.
- **Empty candle array** — `runBacktest` guards at entry and returns a zero-trade `BacktestResult` with `reason: 'no-data'`. The table row shows "No data."
- **Preset produces zero signals over the window** — valid result, not an error. Shown as "0 trades" with flat equity curve.

## Testing

- **Unit — entry modules.** Each of the 4 entry modules gets a test in `packages/strategies/__tests__/entry/<id>.test.ts` using fixture candles. Tests verify signal count, direction, and key decision points.
- **Unit — `runBacktest`.** Deterministic: same candles + preset → identical result. One test per preset.
- **Unit — presets registry.** All 5 presets resolve, entry modules are callable, allocation/risk configs match known enum values.
- **Integration — preset snapshot.** 5 presets × 1 symbol × 100-bar fixture → JSON snapshot of the metrics table. Catches accidental logic drift across the whole pipeline.
- **Integration — default preset parity.** The `hmm-top3` preset run against a snapshot of today's signal log must match the log's signals 1:1. Guards production behavior.
- **E2E (Playwright) — `/backtest` comparison.** Load page, select 3 presets, verify equity chart renders 3 lines, verify metrics table has 3 rows, verify "best" highlighting is applied per column.
- **No changes to existing live-signal tests.** Production code path is unchanged modulo the preset dispatch, which is covered by unit + integration tests.

## Rollout

1. Land `packages/strategies/` with types, presets, and the 4 entry modules wrapping existing code. `hmm-top3` must match production signal log (integration test gate).
2. Add `vwap-ema-bb` entry module + unit tests.
3. Extract `runBacktest` from `backtest/page.tsx`; replace inline call with `runBacktest(candles, hmm-top3Preset)`. No UI change yet. Verify existing backtest behavior unchanged.
4. Add comparison UI: multi-select, overlay chart, metrics table.
5. Wire live path: `SIGNAL_ENGINE_PRESET` env var, `strategyId` on signal records, UI badge.
6. E2E test + docs.

Each step is independently shippable and reviewable.
