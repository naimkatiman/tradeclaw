# TradeClaw Pilot — Binance Futures Auto-Execution + MetaApi Bridge

Status: Phase 1 in progress
Owner: Zaky
Started: 2026-05-01
Last updated: 2026-05-02

## What this is

A new product surface on tradeclaw.win called **TradeClaw Pilot** that takes the
production trend-following signals already generated by `signal-generator.ts`
and executes them on the user's own Binance USDⓈ-M Futures account (Phase 1
and 2), and later on MetaApi-connected MT5 brokers for FX, indices, and metals
(Phase 3).

Pilot sits one tier above the existing $29/mo Signals product, priced at
$99/mo or $990/yr. Users authorize their own broker keys; we never custody
funds.

## Why this is its own product, not a feature flag

- Different value prop: signals → autonomous execution. Different SLA, support
  surface, and risk profile.
- Different cost structure: per-user broker calls, monitoring, and incident
  response cost more per user than alert delivery.
- Different compliance posture: auto-execution of leveraged trades demands
  explicit terms, disclaimers, and an audit trail. Bolting that into the
  existing Signals tier muddies both products.
- Different failure mode: a broken signal alert is annoying. A broken
  executor is account-draining. We need separate kill switches, separate
  on-call rules, and a separate dashboard.

## Strategy: trend-following with hard risk rails

There is no guaranteed alpha. The defensible edge for retail leveraged
futures is strict trend-following with risk management that survives losing
streaks. Win rate sits at 40 to 50 percent; profitability comes from average
winner > average loser and from never blowing up a tail.

### Entry filters (all must pass)

- Source signal: `signal_history` row with `strategy_id='hmm-top3'`,
  `created_at` within the last 5 minutes, no existing `executions` row.
- Direction filter: H1 EMA-50 slope agrees with signal direction.
  Long only if H1 EMA-50 rising; short only if falling.
- Regime filter: HMM regime from hmm-top3 must not be `ranging`, AND
  ADX(14) on H1 must be ≥ 20.
- Universe filter: symbol must be in today's universe snapshot (see below).
- Concurrency: no existing open position on the same symbol. Pyramiding
  disabled in v1.
- Concurrency global: max 4 open positions per account.

### Order construction

- Entry: market order, reduceOnly=false.
- Stop loss: STOP_MARKET, reduceOnly=true, distance = 1.5 × ATR(14) on
  signal timeframe.
- Take profit: split.
  - TP1 at 1.5R, half position, TAKE_PROFIT_MARKET reduceOnly=true.
  - On TP1 fill: move SL to breakeven, leave runner with chandelier exit
    (3 × ATR trailing). Implemented via a per-minute stop-update job.
- All orders carry `clientOrderId = signal_id` for idempotency.

### Sizing

- Risk per trade = 1% of current account equity. On $1000 testnet that is $10.
- Position notional = risk_usd × entry_price / stop_distance_usd.
- Implied leverage = notional / margin. Capped at **5x**. If implied > 5x,
  shrink notional, never widen the stop.
- Margin type: ISOLATED. Position mode: one-way.

### Risk rails (mandatory before any live trade)

- Daily loss kill: realized PnL ≤ -5% of starting daily equity → halt new
  entries until next UTC day. Open positions keep stops.
- Weekly loss kill: -12% → halt all activity, require manual reset via
  admin route.
- Per-trade cap: notional ≤ 25% of equity even if sizing math says higher.
- Symbol cap: max 1 position per symbol.
- Kill switch env: `EXECUTION_MODE=disabled` halts all order placement
  immediately, no deploy required.

## Symbol universe — dynamic, not static

Static allowlists rot. We screen daily.

### Three quantitative gates

1. **Liquidity**: 24h notional volume ≥ $100M on Binance Futures.
2. **Trendiness**: 20-day Efficiency Ratio
   = abs(close_today - close_20d_ago) / sum(daily absolute moves)
   must be ≥ 0.30.
3. **Correlation dedup**: cluster remaining symbols by 30-day return
   correlation. Within each cluster (correlation > 0.85), keep only the
   highest-volume member.

Output: 10 to 20 symbols, refreshed daily at 00:05 UTC. Symbols dropping
out keep open positions; new entries blocked. Symbols entering become
eligible immediately.

Persisted in `universe_snapshots` (date, symbol, ef_ratio, vol_24h_usd,
included). Daily history retained for backtest and audit.

### Expected starting universe (rough estimate, regenerated by cron)

BTCUSDT, ETHUSDT, SOLUSDT, BNBUSDT, DOGEUSDT, AVAXUSDT, LINKUSDT, SUIUSDT,
ARBUSDT, OPUSDT, LTCUSDT, APTUSDT.

## Architecture

### Execution flow

```
[signal_generator writes signal_history row]
                │
                ▼
[/api/cron/execute runs every 60s]
                │
                ├─ pulls unexecuted hmm-top3 rows from last 5 min
                ├─ applies entry filters (direction, regime, universe, concurrency)
                ├─ for each survivor:
                │     ├─ fetch ATR, compute stop distance
                │     ├─ fetch equity, compute size + leverage
                │     ├─ place bracket via Binance client
                │     │     (entry market, SL stop_market, TP1 take_profit_market)
                │     ├─ write executions row (status='pending'|'filled'|'rejected')
                │     └─ on reject, write execution_errors row
                ▼
[/api/cron/manage-positions runs every 60s]
                │
                ├─ for each open position with TP1 filled:
                │     ├─ move SL to breakeven (idempotent, tagged via clientOrderId)
                │     └─ recompute chandelier trail, update SL if widened past breakeven
                ▼
[/api/cron/universe runs daily 00:05 UTC]
                │
                └─ recompute universe_snapshots
```

### Why pull, not push

The fire-and-forget hook from `tracked-signals.ts` is wrong for execution.
- Multi-instance dupes: two Railway instances would both place the same trade.
- Hot path latency: trading the order placement onto an API request adds
  variable latency to user traffic.
- Hard to retry on transient failure.

Pull from cron with row-level lock is multi-instance safe and naturally
retries on the next tick.

## Schema additions (migration 018)

### executions

```sql
CREATE TABLE executions (
  id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  signal_id       UUID NOT NULL REFERENCES signal_history(id),
  user_id         UUID,                          -- NULL in Phase 1 (house account)
  broker          VARCHAR(32) NOT NULL,          -- 'binance-futures' | 'metaapi'
  mode            VARCHAR(16) NOT NULL,          -- 'testnet' | 'live'
  symbol          VARCHAR(32) NOT NULL,
  side            VARCHAR(8)  NOT NULL CHECK (side IN ('BUY','SELL')),
  qty             NUMERIC(24,12) NOT NULL,
  entry_price     NUMERIC(24,12),
  stop_price      NUMERIC(24,12),
  tp1_price       NUMERIC(24,12),
  leverage        SMALLINT NOT NULL,
  notional_usd    NUMERIC(18,4),
  risk_usd        NUMERIC(18,4),
  client_order_id VARCHAR(64) NOT NULL UNIQUE,
  exchange_order_id VARCHAR(64),
  status          VARCHAR(16) NOT NULL,          -- pending|filled|partially_filled|rejected|closed
  filled_at       TIMESTAMPTZ,
  closed_at       TIMESTAMPTZ,
  realized_pnl    NUMERIC(18,4),
  created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

### execution_errors

```sql
CREATE TABLE execution_errors (
  id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  signal_id    UUID,
  execution_id UUID,
  user_id      UUID,
  broker       VARCHAR(32) NOT NULL,
  stage        VARCHAR(32) NOT NULL,             -- size|place_entry|place_sl|place_tp|manage|cancel
  error_code   VARCHAR(64),
  error_msg    TEXT NOT NULL,
  payload      JSONB,
  created_at   TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

### universe_snapshots

```sql
CREATE TABLE universe_snapshots (
  snapshot_date DATE NOT NULL,
  symbol        VARCHAR(32) NOT NULL,
  ef_ratio      NUMERIC(8,4) NOT NULL,
  vol_24h_usd   NUMERIC(20,2) NOT NULL,
  included      BOOLEAN NOT NULL,
  excluded_reason VARCHAR(64),                   -- liquidity|trendiness|correlation
  PRIMARY KEY (snapshot_date, symbol)
);
```

## Environment variables

### Phase 1 (house testnet account)

```
BINANCE_API_KEY            <set on Railway, web service>
BINANCE_API_SECRET         <set on Railway, web service>
BINANCE_BASE_URL           https://testnet.binancefuture.com  # signed orders here
BINANCE_MARKET_DATA_URL    https://fapi.binance.com           # klines/ticker here
EXECUTION_MODE             disabled    # flip to testnet after dry-run handshake
EXEC_RISK_PCT              1
EXEC_MAX_LEVERAGE          5
EXEC_MAX_POSITIONS         4
EXEC_DAILY_LOSS_PCT        5
EXEC_WEEKLY_LOSS_PCT       12
EXEC_PER_TRADE_NOTIONAL_PCT 25
EXEC_TELEGRAM_CHAT_ID      <Zaky private chat id>
```

**Why the URL split.** Testnet has synthetic 24h volumes (DOGE = $125B, BTC = $108B
with ER=0.22) and sparse klines, so screening on testnet data picks junk
symbols and the executor's ATR/EMA/ADX filters become noise. Splitting routes
unsigned reads (klines, markPrice, 24h ticker) to mainnet while signed orders
+ exchangeInfo (LOT_SIZE filters that must match the trading venue) stay on
testnet. Universe becomes (testnet TRADING list) ∩ (mainnet liquid + trending),
so the soak measures real strategy expectancy instead of testnet noise. Single-
URL deploys (mainnet-only or testnet-only) keep working — `BINANCE_MARKET_DATA_URL`
falls back to `BINANCE_BASE_URL` when unset.

## Risks and mitigations

| Risk | Mitigation |
|---|---|
| Liquidation cascade on leveraged loser | ISOLATED margin + 1.5×ATR stop + 5x leverage cap + daily/weekly kill switches |
| Multi-instance executes same signal twice | clientOrderId = signal_id (Binance rejects duplicate); pull-based cron with row-level lock |
| signal_history schema change breaks executor | Foreign key on signal_id; integration test in CI exercises full path |
| Universe screen excludes everything in a chop month | Min-floor: if screen returns < 5 symbols, fall back to BTCUSDT + ETHUSDT only |
| Funding rate eats trend profits | Funding cost > X% of position notional → skip entry. Monitor funding via /fapi/v1/premiumIndex |
| User pastes wrong key permission scope (e.g. spot only) | Handshake on key save validates futures permission via /fapi/v2/account |
| Signal bug ships to production, every Pilot user loses money simultaneously | Phase 2 multi-tenant rollout is staged: 5 beta → 25 → 100 → public. Killswitch at every stage |
| Telegram outage masks fill/error stream | Errors also logged to execution_errors table + admin dashboard |
| MetaApi outage takes down MT5 fills | Phase 3 only. Show clear "broker degraded" banner; SL still managed by MT5 itself |

## Phase 1.x deferred work (knowingly skipped in 1.0)

Items below are listed in earlier sections as if implemented; they are not.
Captured here so reviewers don't think they were forgotten.

- **Daily / weekly loss kill switches.** `EXEC_DAILY_LOSS_PCT=5` and
  `EXEC_WEEKLY_LOSS_PCT=12` are read by no code today. Acceptable risk on a
  $1000 testnet because the worst case is bankrolling test losses, but
  **mandatory before any live mainnet flip**. Implementation: query
  `executions` for `realized_pnl` over the last 24h / 7d at the start of
  each `runExecutorTick`, compare against starting equity, halt new entries.
- **`killswitch.ts` (DB-backed kill state).** Plan §architecture lists this
  module; today only the env-var gate `EXECUTION_MODE=disabled` exists.
  Single-tenant Phase 1 doesn't need DB-backed kill — env flip on Railway
  is fast enough. Required for Phase 2 multi-tenant so admins can halt one
  user without redeploying.
- **PnL backfill on close.** Position manager marks `status='closed'` but
  leaves `realized_pnl=NULL`. Phase 1.5 will diff entry/exit fills and
  populate.
- **Chandelier trail.** Phase 1.0 only moves SL to breakeven on TP1 fill;
  the runner sits at breakeven with no trailing stop. Phase 1.5 will add
  `max(highestHigh − 3×ATR, breakeven)` recomputed each tick.
